# Bugfix Release v1.4.1 Design Spec

## Summary

Fixes 4 user-reported bugs from the v1.4.0 release. All fixes target the M2/M1 codebase on `main` (M3 multi-wallet was reverted in `6a6185d`).

## Bugs Addressed

| # | Bug | Reporter | Severity |
|---|-----|----------|----------|
| 1 | "Mnemonic not found" + skips to dashboard after APK upgrade | Neon | High |
| 2 | Onboarding bypass: close app during mnemonic verification, reopens to dashboard | Radoslav | Critical |
| 3 | Network switch doesn't visibly restart the app | Radoslav | Low/UX |
| 4 | No in-app dark mode toggle | Community | Low/UX |

## Version

- `versionName`: `1.4.1`
- `versionCode`: `5` (must exceed v1.4.0's versionCode of 4 for upgrade path)

**Note:** The M3 revert reset `build.gradle.kts` to `versionName = "1.3.0"` / `versionCode = 4`, but the released APK was v1.4.0. We bump to `1.4.1` / `5` to ensure Android accepts the upgrade.

## Fix 1+2: Atomic Write + Onboarding Gate

Bugs 1 and 2 share root cause: the wallet private key is persisted before mnemonic backup verification completes, and there is no gate to route users back to verification on next launch.

### Root Cause Analysis

**Bug 2 (onboarding bypass):**
- `generateWalletWithMnemonic()` calls `savePrivateKey()` (async `apply()`) then `storeMnemonic()` (separate async `apply()`)
- Both writes complete before the user finishes mnemonic verification
- On next launch, `hasWallet()` checks only for `KEY_PRIVATE_KEY` existence -> returns `true` -> skips to dashboard
- No check for `KEY_MNEMONIC_BACKED_UP` status in the start destination logic

**Bug 1 (mnemonic not found on upgrade):**
Two possible sub-causes:
1. **Race condition**: `savePrivateKey()` and `storeMnemonic()` use separate `apply()` calls. If the app is killed between the two, private key exists but mnemonic doesn't.
2. **EncryptedSharedPreferences MasterKey mismatch**: On APK upgrade, if the MasterKey configuration changed (e.g., StrongBox availability differs between devices/versions), the encrypted prefs file exists but can't be decrypted. `contains()` returns `true` (entry exists in XML) but `getString()` returns null or throws.

### Changes

#### 1a. Atomic key+mnemonic write

**File:** `KeyManager.kt` -> `generateWalletWithMnemonic()`

Replace the two separate `edit().apply()` calls with a single `edit().commit()`:

```kotlin
fun generateWalletWithMnemonic(
    wordCount: MnemonicManager.WordCount = MnemonicManager.WordCount.TWELVE
): Pair<WalletInfo, List<String>> {
    val words = mnemonicManager.generateMnemonic(wordCount)
    val privateKeyBytes = mnemonicManager.mnemonicToPrivateKey(words)
    val hex = Numeric.toHexStringNoPrefixZeroPadded(BigInteger(1, privateKeyBytes), 64)

    // Single atomic write - commit() is synchronous, all-or-nothing
    prefs.edit()
        .putString(KEY_PRIVATE_KEY, hex)
        .putString(KEY_WALLET_TYPE, WALLET_TYPE_MNEMONIC)
        .putString(KEY_MNEMONIC, words.joinToString(" "))
        .putBoolean(KEY_MNEMONIC_BACKED_UP, false)
        .commit()

    return Pair(getWalletInfo(), words)
}
```

Apply the same pattern to `importWalletFromMnemonic()`:

```kotlin
fun importWalletFromMnemonic(
    words: List<String>,
    passphrase: String = ""
): WalletInfo {
    require(mnemonicManager.validateMnemonic(words)) { "Invalid mnemonic" }

    val privateKeyBytes = mnemonicManager.mnemonicToPrivateKey(words, passphrase)
    val hex = Numeric.toHexStringNoPrefixZeroPadded(BigInteger(1, privateKeyBytes), 64)

    // Single atomic write
    prefs.edit()
        .putString(KEY_PRIVATE_KEY, hex)
        .putString(KEY_WALLET_TYPE, WALLET_TYPE_MNEMONIC)
        .putString(KEY_MNEMONIC, words.joinToString(" "))
        .putBoolean(KEY_MNEMONIC_BACKED_UP, false)
        .commit()

    return getWalletInfo()
}
```

After this refactor, the private methods `savePrivateKey()` and `storeMnemonic()` are still used by `generateWallet()` and `importWallet()` (raw-key paths). `storeMnemonic()` becomes dead code and should be removed.

#### 1b. Onboarding completion gate

**File:** `MainActivity.kt` -> `onCreate()`

Add a check between `hasWallet()` and the PIN/Main routes:

```kotlin
val startDestination = when {
    !repository.hasWallet() -> Screen.Onboarding.route
    repository.needsMnemonicBackup() -> Screen.MnemonicBackup.route
    pinManager.hasPin() -> Screen.Auth.route
    else -> Screen.Main.route
}
```

**File:** `GatewayRepository.kt`

Add convenience method:

```kotlin
fun needsMnemonicBackup(): Boolean {
    return keyManager.getWalletType() == KeyManager.WALLET_TYPE_MNEMONIC
        && !keyManager.hasMnemonicBackup()
}
```

This ensures users who closed the app mid-backup are routed back to complete it.

**Note:** When `MnemonicBackupScreen` is the start destination, pressing back closes the app (empty back stack). This is intentional — the user should not be able to skip backup by pressing back.

#### 1c. EncryptedSharedPreferences crash guard

**File:** `KeyManager.kt` -> `encryptedPrefs` lazy initializer

```kotlin
private var walletResetDueToCorruption = false

private val encryptedPrefs: SharedPreferences by lazy {
    try {
        createEncryptedPrefs(useStrongBox = true)
    } catch (e: Exception) {
        Log.e("KeyManager", "EncryptedSharedPreferences corrupted, resetting", e)
        // Delete the corrupted file and retry without StrongBox
        // (StrongBox availability may differ between device/version upgrades)
        context.deleteSharedPreferences("ckb_wallet_keys")
        walletResetDueToCorruption = true
        try {
            createEncryptedPrefs(useStrongBox = false)
        } catch (e2: Exception) {
            // Last resort: try with StrongBox again on clean slate
            createEncryptedPrefs(useStrongBox = true)
        }
    }
}

fun wasResetDueToCorruption(): Boolean = walletResetDueToCorruption

private fun createEncryptedPrefs(useStrongBox: Boolean = true): SharedPreferences {
    val masterKey = MasterKey.Builder(context)
        .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
        .setRequestStrongBoxBacked(useStrongBox)
        .build()

    return EncryptedSharedPreferences.create(
        context,
        "ckb_wallet_keys",
        masterKey,
        EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
        EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM
    )
}
```

If the MasterKey changed between versions and the prefs can't be decrypted, this wipes the corrupted file and retries **without StrongBox** (since StrongBox flakiness across device upgrades is a known Android issue). A `walletResetDueToCorruption` flag is exposed so the Onboarding screen can show a one-time warning: "Your previous wallet data was corrupted and had to be reset. If you have a backup, please import it."

The user loses their wallet data but sees Onboarding (can re-import) instead of a broken dashboard.

## Fix 3: Network Switch Restart (ProcessPhoenix pattern)

### Root Cause

`Process.killProcess(myPid())` kills the process but doesn't guarantee Android relaunches the app. On some devices/launchers the app just disappears. Users expect a visible close-and-reopen.

### Change

**File:** `GatewayRepository.kt` -> `switchNetwork()`

Replace bare `Process.killProcess()` with Intent-based restart:

```kotlin
suspend fun switchNetwork(target: NetworkType): Result<Unit> = runCatching {
    if (target == currentNetwork) return@runCatching
    if (_isSwitchingNetwork.value) throw Exception("Network switch already in progress")

    _isSwitchingNetwork.value = true
    try {
        cacheManager.clearAll()
        daoSyncManager.clearAll()
        walletPreferences.setSelectedNetwork(target)

        // ProcessPhoenix-style restart: launch fresh, then exit
        val intent = context.packageManager.getLaunchIntentForPackage(context.packageName)!!
        intent.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK or Intent.FLAG_ACTIVITY_CLEAR_TASK)
        context.startActivity(intent)
        // Use killProcess instead of Runtime.exit(0) — exit(0) runs shutdown hooks
        // which may call nativeStop() and block indefinitely on JNI peer disconnection
        android.os.Process.killProcess(android.os.Process.myPid())
    } catch (e: Exception) {
        _isSwitchingNetwork.value = false
        throw e
    }
}
```

No third-party dependency. The Intent starts a fresh MainActivity before the process dies, so the user sees the app reopen immediately.

## Fix 4: Dark Mode In-App Toggle

### Root Cause

Dark mode works via `isSystemInDarkTheme()` but there's no in-app toggle. Users with system light mode have no way to use dark mode in the app.

### Changes

#### 4a. ThemeMode enum + preference

**File:** `WalletPreferences.kt`

```kotlin
enum class ThemeMode { SYSTEM, LIGHT, DARK }

fun getThemeMode(): ThemeMode {
    val name = prefs.getString(KEY_THEME_MODE, ThemeMode.SYSTEM.name)
    return try {
        ThemeMode.valueOf(name ?: ThemeMode.SYSTEM.name)
    } catch (e: IllegalArgumentException) {
        ThemeMode.SYSTEM
    }
}

fun setThemeMode(mode: ThemeMode) {
    prefs.edit().putString(KEY_THEME_MODE, mode.name).apply()
}

companion object {
    // ... existing keys
    private const val KEY_THEME_MODE = "theme_mode"
}
```

#### 4b. Update CkbWalletTheme

**File:** `Theme.kt`

Add a `themeMode` parameter that resolves the dark/light boolean:

```kotlin
@Composable
fun CkbWalletTheme(
    themeMode: ThemeMode = ThemeMode.SYSTEM,
    dynamicColor: Boolean = false,
    content: @Composable () -> Unit
) {
    val darkTheme = when (themeMode) {
        ThemeMode.SYSTEM -> isSystemInDarkTheme()
        ThemeMode.LIGHT -> false
        ThemeMode.DARK -> true
    }
    // ... rest unchanged
}
```

#### 4c. Reactive theme state via StateFlow

**File:** `WalletPreferences.kt`

Add a `StateFlow` for reactive theme updates:

```kotlin
private val _themeMode = MutableStateFlow(getThemeMode())
val themeModeFlow: StateFlow<ThemeMode> = _themeMode.asStateFlow()

fun setThemeMode(mode: ThemeMode) {
    prefs.edit().putString(KEY_THEME_MODE, mode.name).apply()
    _themeMode.value = mode
}
```

#### 4d. Wire through MainActivity

**File:** `MainActivity.kt`

Inject `WalletPreferences` directly (it's already a `@Singleton` Hilt-provided object — no need to go through `GatewayRepository` which has it as a private field):

```kotlin
@Inject
lateinit var walletPreferences: WalletPreferences

// In setContent:
val themeMode by walletPreferences.themeModeFlow.collectAsState()

CkbWalletTheme(themeMode = themeMode) { ... }
```

This is reactive — when `setThemeMode()` is called from SettingsScreen, the StateFlow emits and the theme updates immediately without restart.

#### 4e. Settings screen toggle

**File:** `SettingsScreen.kt` / `SettingsViewModel.kt`

Add a "Theme" row showing the current mode. On tap, show a dialog with three options: System, Light, Dark. SettingsViewModel exposes `themeMode: StateFlow<ThemeMode>` (collected from `WalletPreferences.themeModeFlow`) and a `setThemeMode(mode)` action that delegates to `WalletPreferences.setThemeMode()`.

## M3 Forward Compatibility

These fixes are on `main` (M2 codebase). When M3 is re-merged (revert-the-revert), the following must be addressed:

### Checklist before re-merging M3

1. **Atomic write in wallet-scoped KeyManager**: `storeKeysForWallet()` in M3 has the same race condition (separate `apply()` calls for key and mnemonic). Apply the same single-`commit()` pattern.

2. **WalletMigrationHelper must copy backup flag**: Currently M3's migration helper copies private key and mnemonic from legacy prefs to wallet-scoped prefs, but does NOT copy `KEY_MNEMONIC_BACKED_UP`. Add this to the migration.

3. **Onboarding gate for multi-wallet**: The gate checks `needsMnemonicBackup()` which uses the legacy single-wallet KeyManager. In M3, this needs to check the **active wallet's** backup status via `WalletRepository`.

4. **Crash guard on wallet-scoped prefs**: The `createEncryptedPrefs()` crash guard pattern needs to be applied to `getWalletPrefs(walletId)` in M3 as well.

5. **Theme preference**: `ThemeMode` is stored in `WalletPreferences` (non-encrypted SharedPreferences). M3 doesn't touch this — carries forward cleanly.

6. **ProcessPhoenix restart**: M3 uses the same `switchNetwork()`. Carries forward cleanly.

7. **Version**: M3 release should be >= v1.5.0 / versionCode >= 6.

### Upgrade path: v1.4.0 -> v1.4.1 -> M3

- **v1.4.0 -> v1.4.1**: Users get bugfixes. If their prefs were corrupted, crash guard wipes and they re-import. If they had incomplete backup, onboarding gate sends them to verify.
- **v1.4.1 -> M3**: WalletMigrationHelper runs, copies all keys + mnemonic + backup flag to wallet-scoped storage. Theme preference persists. Network switch uses ProcessPhoenix.

## Testing Plan

1. **Fresh install**: Create wallet -> verify backup flow completes end-to-end
2. **Kill during backup**: Create wallet -> close app on step 2 -> reopen -> should land on MnemonicBackupScreen
3. **Upgrade install**: Install v1.4.0 APK -> upgrade to v1.4.1 -> verify wallet data intact, mnemonic accessible
4. **Corrupted prefs**: Simulate by modifying the encrypted prefs XML -> app should wipe and show Onboarding
5. **Network switch**: Switch mainnet/testnet -> app should visibly close and reopen on new network
6. **Dark mode**: Toggle through SYSTEM/LIGHT/DARK -> theme applies immediately
7. **Import via private key**: Import raw key -> backup screen should say "no mnemonic" (expected, not a bug for raw key wallets) -> should NOT be gated by onboarding completion (only mnemonic wallets are gated)
